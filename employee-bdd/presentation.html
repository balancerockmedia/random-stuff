Test Driven Development

A software development technique that uses short cycles to incrementally design and develop software from the outside in


  1. Write a test that shows the intent of what you would like the new code to do
  2. Write the code to make the test pass
  3. Refactor if needed to make it better

Why write tests first?

  • Forces you to think about the code before writing it
  • Helps prevent from over-engineering the solution
  • Write only the code that you need
  • Reduces the ownership that any one developer has on a section of code
  • Leads to better code coverage and less regressions
  • Sometimes you can't write tests first and that's ok

Behavior Driven Development

"Many software projects suffer from low-quality communication between the domain experts and programmers on the team"


  • Merges TDD with automated acceptance testing
  • A common language is needed to express the acceptance tests in a way that the stakeholder and the developer understand
  • Similar to how a domain model can help everybody to understand the business objects
  • Acceptance tests written as examples of the functionality that everybody can understand
  • They become the living documentation
  • Broken down into features, scenarios and steps

Cucumber

  • Command line tool that was first built for Ruby, but now has support for many other languages including JavaScript
  • Reads specifications from plain-language text files called features, examines them for scenarios to test, and then runs the scenarios against your system
  • Can be easily tied into build tools such as Grunt and Jenkins

Example Feature


Feature: Sign up
  Sign up should be quick and friendly.
    
  Scenario: Successful sign up
    
    New users should get a confirmation email and be 
    greeted personally by the site once signed in.
        
      Given I have chosen to sign up
      When I sign up with valid details
      Then I should receive a confirmation email
      And I should see a personalized greeting message

Example Feature


Feature: Employee

  Scenario: Create an employee and change their title
  
    Given an employee with the name of "Dan" and a title of "associate"
    Then their name should be "Dan"
    And their title should be "associate"
    When I can change their title to "senior associate"
    Then their new title should be "senior associate"

Example Step Deffinitions


this.Given(
  /^an employee with the name of "([^"]*)" and a title of "([^"]*)"$/,
  function(arg1, arg2, callback) {
    // code to implement this step
    callback();
  }
);

this.Then(/^their name should be "([^"]*)"$/, function(arg1, callback) {
  // code to implement this step
  callback();
});

this.Then(/^their title should be "([^"]*)"$/, function(arg1, callback) {
  // code to implement this step
  callback();
});

this.When(/^I can change their title to "([^"]*)"$/, function(arg1, callback) {
  // code to implement this step
  callback();
});

this.Then(/^their new title should be "([^"]*)"$/, function(arg1, callback) {
  // code to implement this step
  callback();
});
